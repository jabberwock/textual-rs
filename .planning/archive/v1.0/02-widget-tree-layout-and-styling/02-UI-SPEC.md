# Phase 2 — UI Design Contract

> Visual and interaction contract for the terminal UI rendered by textual-rs Phase 2.
> This is a pure Rust TUI library — there is no web frontend, no shadcn, no Tailwind.
> "UI" here means the terminal cell output: colors as ANSI/RGB ratatui Color values,
> spacing as cell counts, typography as ratatui text styles, and interactions as
> keyboard/focus state changes in the terminal.
>
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

## Spacing Scale

Terminal spacing is measured in **cells** (columns or rows), not pixels. The standard 8-point grid maps directly to cells: where the 8-point system uses 8px as the base unit, this contract uses 1 cell as the atomic unit. The spacing token scale therefore contains only values that are multiples of 4 cells — matching the 8-point grid's requirement that all spacing tokens be multiples of 4. Sub-4-cell values (1 cell, 2 cells, 3 cells) are not ambiguous layout choices; they are fixed structural dimensions dictated by ratatui's border model and dock bar mechanics. Those values are captured separately as **structural constants** and are explicitly outside the token system.

### Spacing Tokens (multiples of 4 cells only)

| Token | Cell Value | Usage |
|-------|-----------|-------|
| sm | 4 cells | Default padding for content areas; gap between sidebar and main panel |
| md | 8 cells | Major layout region gap; outer spacing between large sections |
| lg | 16 cells | Reserved — not used in Phase 2 demo; available for future full-screen outer margins |

### Structural Constants (not tokens — fixed by terminal/ratatui mechanics)

These values are architectural constraints, not design choices. They must not be used as spacing tokens in TCSS or style calculations.

| Constant | Cell Value | Reason |
|----------|-----------|--------|
| Border width | 1 cell (each side) | ratatui border consumes exactly 1 cell per side; this is a rendering constraint, not a spacing token |
| Title/footer dock bar height | 1 row | Minimum usable dock bar; 1 row for Header/Footer with no border (CONTEXT.md D-06) |
| Input bar height (with border) | 3 rows | 1 content row + 1 top border + 1 bottom border; border cost is structural, not spacing |

**Notes:**

- Widget borders consume 1 cell on each side — account for this when computing inner content area.
- Mouse hit map resolution: 1 cell = 1 unit (col × row). Source: RESEARCH.md LAYOUT architecture.

## Typography

Terminal typography uses ratatui `Style` modifiers. There is no variable font size — all text renders at 1 cell height in the user's monospace terminal font. Differentiation is achieved through **weight (bold)**, **emphasis (italic/dim)**, and **color**.

| Role | ratatui Modifier | Color Role | Line Height | Usage |
|------|-----------------|------------|-------------|-------|
| Body | `Modifier::empty()` (normal) | Dominant foreground | 1 row/line | Widget content, log output, list items |
| Label | `Modifier::BOLD` | Dominant foreground | 1 row/line | Widget titles, border titles, headings |
| Muted | `Modifier::DIM` | Secondary foreground | 1 row/line | Placeholder text, disabled state, metadata |
| Accent | `Modifier::BOLD` | Accent color | 1 row/line | Focused elements, selected items, CTAs |

**Declared weights (2 maximum):**
- Regular: `Modifier::empty()` — default body text
- Bold: `Modifier::BOLD` — labels, titles, focused/selected state

**Italic** is reserved for CSS parse error diagnostics only (displayed in demo output). No other usage.

## Color

All colors are ratatui `Color` values. The 60/30/10 split applies to terminal cell foreground/background coverage across the rendered screen.

| Role | Value | Coverage | Usage | Source |
|------|-------|----------|-------|--------|
| Dominant (60%) | `Color::Reset` (terminal default background) | Main content areas, widget interiors | Background for most cells; defers to user's terminal theme | Default — terminal portability |
| Secondary (30%) | `Color::Rgb(30, 30, 46)` (dark surface, Catppuccin Mocha base) | Sidebar panels, header/footer bars, border fills | Visually distinct regions in dock layout demo | Design default |
| Accent (10%) | `Color::Rgb(137, 180, 250)` (Catppuccin Mocha Blue) | Focus ring, selected item highlight, active tab indicator, primary CTA text | Accent reserved for: focused widget border, selected list row background, active tab label bold | Design default |
| Destructive | `Color::Rgb(243, 139, 168)` (Catppuccin Mocha Red) | Destructive confirmation prompts only | CSS parse error indicators in demo output | Design default |

**Foreground text colors:**

| Role | Value | Usage |
|------|-------|-------|
| Primary text | `Color::Rgb(205, 214, 244)` (Catppuccin Mocha Text) | Body copy, labels |
| Muted text | `Color::Rgb(108, 112, 134)` (Catppuccin Mocha Overlay0) | Placeholder, disabled, metadata |
| Inverted text | `Color::Rgb(30, 30, 46)` (Catppuccin Mocha Base) | Text on accent-colored backgrounds |

**Rationale:** Catppuccin Mocha is the design default for the Phase 2 demo. DEFAULT_CSS for built-in widgets uses `Color::Reset` (transparent) so user stylesheets fully override colors. The Catppuccin palette is only the concrete reference for the layout demo output verification.

**Accent reserved for (explicit list):**
1. Focused widget border — when `:focus` pseudo-class is active, border color switches to accent
2. Selected list row — background color of the highlighted item in a scrollable list
3. Active tab label — bold + accent color for the currently active tab
4. Screen title text in the Header bar

## Copywriting Contract

This is a Rust library with no user-facing web copy. The "copywriting" contract governs the **demo application text** (the IRC-client layout demo per CONTEXT.md specifics) and **diagnostic/error messages** emitted by the framework itself.

| Element | Copy | Source |
|---------|------|--------|
| Primary CTA (demo) | "Send Message" (input bar submit button label in IRC demo) | CONTEXT.md specifics: IRC client demo; revised from "Send" per checker flag — verb + noun required |
| Empty state — no widgets mounted | "No widgets mounted. Call compose() to add children." | Default |
| Empty state — empty list widget | "(empty)" | Default — matches Python Textual convention |
| CSS parse error — unknown property | "Unknown property '{name}' at line {n} — ignored" | Default |
| CSS parse error — invalid value | "Invalid value for '{property}': '{value}' at line {n} — using default" | Default |
| CSS parse error — malformed selector | "Malformed selector at line {n} — rule skipped" | Default |
| Unmount of non-existent widget | "WidgetId {id} not found in arena — unmount is a no-op" | Default |
| Focus on non-focusable widget | No message — silently skip to next focusable widget in tab order | CONTEXT.md D-14 |

**Destructive actions in Phase 2:**

| Action | Trigger | Confirmation Approach |
|--------|---------|----------------------|
| `unmount()` subtree removal | API call — removes widget + all descendants from arena | No runtime confirmation — API is synchronous and immediate. Callers are responsible. Document as "irreversible" in rustdoc. |
| Screen pop | `app.pop_screen()` API | No runtime confirmation — same pattern as `unmount()`. |

No interactive destructive confirmation dialogs exist in Phase 2 (no modal system yet — screen push/pop is just arena manipulation).

## Interaction States

All interaction state changes are expressed through CSS pseudo-classes updating the `ComputedStyle` for affected widgets. This section specifies the visual output for each state.

| State | CSS Pseudo-class | Visual Change | Trigger |
|-------|-----------------|---------------|---------|
| Default | (none) | Widget renders with DEFAULT_CSS colors | Widget is mounted |
| Focused | `:focus` | Border color → accent (`Color::Rgb(137, 180, 250)`); border style unchanged | Tab key or `AppContext::set_focus()` |
| Hovered | `:hover` | Border color → secondary foreground (dim). Not interactive in Phase 2 — layout demo only | Mouse move event over widget Rect (Phase 2 wires hit map; Phase 3 adds dispatch) |
| Disabled | `:disabled` | All text → muted (`Modifier::DIM`); border → muted foreground | `Widget::set_disabled()` API |
| Dirty (layout) | (internal) | No visual change — dirty flag triggers relayout only | Widget state change; resize event |

**Tab focus behavior:**
- Tab moves focus to next `can_focus() == true` widget in depth-first DOM order (CONTEXT.md D-14)
- Shift+Tab moves to previous (same order, reversed)
- If no widget has focus, Tab focuses the first focusable widget in the active screen
- Focus wraps: last focusable → first focusable on Tab; first → last on Shift+Tab

## Border Styles

Border style is the primary visual structural element in a terminal UI. Phase 2 implements the full TCSS border-style property set (CSS-08).

| TCSS Value | ratatui BorderType | Usage |
|------------|-------------------|-------|
| `solid` | `BorderType::Plain` | Default for most containers |
| `rounded` | `BorderType::Rounded` | Panels, modals, focused input areas |
| `heavy` | `BorderType::Thick` | High-emphasis containers, selected item borders |
| `double` | `BorderType::Double` | Dialog overlays, important sections |
| `ascii` | `BorderType::Plain` with `+`, `-`, `\|` chars | Fallback for non-Unicode terminals |
| `none` | No border rendered | Content-only areas, transparent containers |

**Default border for all widgets:** `none` (no border unless DEFAULT_CSS or user stylesheet specifies one).
**Focused widget border style:** Inherits the widget's declared border style; only the **color** changes to accent on focus.

## Layout Demo Visual Specification

The Phase 2 success proof is an IRC-client-style layout demo (CONTEXT.md specifics). This section specifies the exact visual structure.

```
┌─ textual-rs demo ───────────────────────────────────────────────────────────┐  <- Header bar, dock: top, 1 row, accent title
│ #general                                                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│ ┌─ Channels ──┐ ┌─ #general ──────────────────────┐ ┌─ Users ────────────┐ │  <- flex-row main region
│ │ #general    │ │ <alice> hello world               │ │ @alice             │ │
│ │ #random     │ │ <bob>   hey there                 │ │ bob                │ │
│ │ #dev        │ │ <charlie> anyone seen the docs?   │ │ charlie            │ │
│ │             │ │                                   │ │                    │ │  <- channel list: fixed 16 cols
│ │             │ │                                   │ │                    │ │  <- user list: fixed 20 cols
│ │             │ │                                   │ │                    │ │  <- chat area: 1fr (flex-grow)
│ └─────────────┘ └───────────────────────────────────┘ └────────────────────┘ │
│ ┌─ Message ───────────────────────────────────────────────────────────────┐  │  <- Input bar, dock: bottom, 3 rows
│ │ Type a message...                            Send Message               │  │
│ └─────────────────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────────────┘
```

**Structural specifics:**

| Region | TCSS | Size |
|--------|------|------|
| Header bar | `dock: top` | 1 row, full width |
| Channel list | `width: 18; border: solid` | Fixed 18 columns |
| Chat area | `flex-grow: 1; border: solid` | Remaining width (1fr) |
| User list | `width: 22; border: solid` | Fixed 22 columns |
| Input bar | `dock: bottom` | 3 rows (1 content + 2 border), full width |

**Context menu overlay (modal test):**
- Push a second Screen with a single `Container` widget
- Container: `width: 40; height: 10; border: rounded` — centered via `position: absolute` + computed inset
- Background: `Color::Rgb(49, 50, 68)` (Catppuccin Mocha Surface0)
- This exercises `push_screen()` / `pop_screen()` and z-ordering
