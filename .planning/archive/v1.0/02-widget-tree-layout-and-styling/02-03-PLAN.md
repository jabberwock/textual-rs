<objective>
Build the TCSS styling engine: cssparser-based tokenizer, hand-rolled selector parser, property parser, cascade resolver with specificity ordering, pseudo-class support, and default CSS mechanism.

Purpose: Widgets need visual styling. The CSS engine parses `.tcss` stylesheets, matches selectors against the widget tree, resolves the cascade (type < class < ID < inline), and produces ComputedStyle for each widget. This feeds into the layout engine (Plan 02-02) which converts ComputedStyle to Taffy layout parameters.

Output: Compiling `css` module with unit tests proving parsing, selector matching, specificity ordering, cascade resolution, pseudo-classes, color parsing, and error collection.
</objective>

    2. Create `crates/textual-rs/src/css/cascade.rs` (per RESEARCH Pattern 6, D-10, D-11):

       ```rust
       use crate::css::parser::Rule;
       use crate::css::selector::{selector_matches, Specificity};
       use crate::css::types::{ComputedStyle, Declaration};
       use crate::widget::WidgetId;
       use crate::widget::context::AppContext;

       #[derive(Debug, Clone)]
       pub struct Stylesheet {
           pub rules: Vec<Rule>,
       }

       impl Stylesheet {
           pub fn parse(css: &str) -> (Self, Vec<String>) {
               let (rules, errors) = crate::css::parser::parse_stylesheet(css);
               (Stylesheet { rules }, errors)
           }
       }
       ```

       Implement `pub fn resolve_cascade(widget_id: WidgetId, stylesheets: &[Stylesheet], ctx: &AppContext) -> ComputedStyle`:
       - Collect all matching rules: for each stylesheet (index=sheet_idx), for each rule (index=rule_idx), find the highest-specificity matching selector
       - Build tuples: `(Specificity, source_order, &Vec<Declaration>)` where source_order = sheet_idx * 100_000 + rule_idx
       - Sort ascending by (specificity, source_order)
       - Apply declarations in sorted order to a `ComputedStyle::default()`
       - Finally apply inline_styles from `ctx.inline_styles[widget_id]` (highest specificity per D-04, D-10)
       - Return computed

       Implement `pub fn apply_cascade_to_tree(screen_id: WidgetId, stylesheets: &[Stylesheet], ctx: &mut AppContext)`:
       - Walk the screen subtree in DFS order
       - For each widget, call resolve_cascade and store result in ctx.computed_styles[id]

       Implement `pub fn collect_default_css(ctx: &AppContext) -> Stylesheet`:
       - This is tricky because `default_css()` is a `where Self: Sized` method, not callable on `dyn Widget`.
       - Approach: default CSS is registered separately. Add a `default_css_registry: HashMap<&'static str, &'static str>` field to a new struct or use a global. Simpler: have each concrete widget type register its default CSS when first mounted. For Phase 2, since we don't have concrete built-in widgets yet, implement a function `pub fn stylesheet_from_css_strings(css_strings: &[&str]) -> (Stylesheet, Vec<String>)` that concatenates CSS strings and parses them. The caller provides default CSS strings at app startup.
       - The Widget trait's `default_css()` method has `where Self: Sized` so it can't be called on `dyn Widget`. This is correct — it's called at registration time (when the concrete type is known), not at runtime through the trait object. For Phase 2, document this pattern and provide the API. Phase 4 (built-in widgets) will register default CSS.
